# STM32WB55 ADC → BLE Notify

A demonstration project that reads the **ADC** value from the **WeAct STM32WB55CGUx** (1 MB Flash) board and broadcasts it via **BLE Notify** — no USB CDC involved.

An Android/iOS phone receives periodic ADC readings through BLE GATT notifications. The value is read in 2 bytes and pushed to the connected device without requiring any write interaction.

```
STM32WB55 ADC Channel
   │  HAL_ADC_Read()
   ▼
Main Loop (periodic)
   │  Advertise + Notify
   ▼
ST BLE Toolbox App (Android / iOS)
```

# Build & Run — Step by Step

## Prerequisites

### Hardware
- WeAct STM32WB55CGUx board (1 MB Flash)
- USB cable (for power only)
- ST-Link V2/V3 programmer

### Software
- STM32CubeIDE
- STM32CubeMX
- STM32CubeProgrammer
- ST BLE Toolbox (Android / iOS)

---

## Step 1 — Flash the Wireless Coprocessor Firmware

The STM32WB55 has a separate Cortex-M0+ core that runs the BLE stack. Its firmware must be flashed **before** building any application.

### 1.1 Flash the FUS (Firmware Upgrade Services)

| Field    | Value                |
|----------|----------------------|
| File     | `stm32wb5x_FUS_fw.bin` |
| Address  | `0x080EE000`         |
| Version  | FUS v2.2.2           |

### 1.2 Flash the BLE Stack

| Field    | Value                          |
|----------|--------------------------------|
| File     | `stm32wb5x_BLE_Stack_full_fw.bin` |
| Address  | `0x080D0000`                   |

### 1.3 Verify

In STM32CubeProgrammer:
- Read **FUS Information**
- Verify **FUS Version** is reported
- Verify **BLE Stack Version** is reported
- Click **Start Wireless Stack**

All four should report success. Disconnect the ST-Link from CubeProgrammer before continuing.

> The wireless binaries can be downloaded from the [STM32CubeWB repository](https://github.com/STMicroelectronics/STM32CubeWB/tree/master/Projects/STM32WB_Copro_Wireless_Binaries/STM32WB5x).

---

## Step 2 — Configure the GATT Service

### 2.1 BLE (STM32_WPAN)

- **Middleware → STM32_WPAN** — enable
  - **BLE Application**
  - **Server Mode**
- Set:
  - `Custom P2P Server` = Disabled
  - `Custom Template` = Enabled

Enabling the Custom Template adds the BLE Advertising and GATT configuration tabs.

### 2.2 Configure BLE Advertising

Open **BLE Advertising** and enable `AD_TYPE_COMPLETE_LOCAL_NAME`. Set the device name:

```
ADC-STM32
```

This is the name that will appear when scanning from a phone.

### 2.3 GATT Service

| Field             | Value          |
|-------------------|----------------|
| Number of Services| `1`            |
| Service Long Name | `ADC_Service`  |
| Service Short Name| `ADC_Service`  |

### 2.4 GATT Characteristic

| Field                       | Value                 |
|-----------------------------|-----------------------|
| Number of Characteristics   | `1`                   |
| Long Name                   | `ADC_Value_Characteristic` |
| Short Name                  | `ADC_Value_Characteristic` |
| Value Length                | `2`                   |

### 2.5 Characteristic Properties

- Enable **NOTIFY** only.
- Disable all other properties (WRITE, READ, etc.).

### 2.6 GATT Events

- Enable `GATT_NOTIFY_ADVERTISING_REPORT_VSEVT` (if needed for debugging)
- Enable `GATT_NOTIFY_WRITE_REQ_AND_EVT_COMPLETE` if you want to react to writes
- Generate the project.

## Step 3 — Modify `custom_stm.h`

In `Core/Inc/custom_stm.h`, inside the `USER CODE BEGIN Includes` block, add:

```c
#include "ble_types.h"
```

This provides the `AD_TYPE_COMPLETE_LOCAL_NAME` constant used in BLE advertising configuration.

---

## Step 4 — Main Loop: Read ADC & Send BLE Notify

All ADC reading and notification logic is implemented in **`Core/Src/main.c`**.

### 4.1 ADC Calibration + Start (USER CODE BEGIN 2)

In `main.c`, after the `MX_*_Init()` calls but before `MX_APPE_Init()`, add:

```c
/* USER CODE BEGIN 2 */
uint32_t lastTick  = HAL_GetTick();

HAL_ADCEx_Calibration_Start(&hadc1, ADC_SINGLE_ENDED);
HAL_ADC_Start_IT(&hadc1);
/* USER CODE END 2 */
```

### 4.2 Periodic Read + Notify in Main Loop (USER CODE BEGIN 3)

In `main.c`, inside the `while (1)` loop, inside the `USER CODE BEGIN 3` block:

```c
/* USER CODE BEGIN 3 */
if (HAL_GetTick() - lastTick >= 1000)
{
    lastTick = HAL_GetTick();

    HAL_ADC_Start(&hadc1);

    if (HAL_ADC_PollForConversion(&hadc1, 10) == HAL_OK)
    {
        uint16_t adcValue = HAL_ADC_GetValue(&hadc1);

        tBleStatus ret = Custom_STM_App_Update_Char(
                        CUSTOM_STM_ADC_VALUE_CHARACTERISTIC,
                        (uint8_t *)&adcValue);
        printf("%u\r\n", adcValue);

        if (ret != BLE_STATUS_SUCCESS)
        {
            printf("BLE Update Failed: 0x%02X\r\n", ret);
        }
    }

    HAL_ADC_Stop(&hadc1);
}
```

### 4.3 How It Works

- `lastTick` tracks the last sample time. Every **1000 ms**, the loop reads one ADC sample.
- `HAL_ADC_Start()` arms the ADC, `HAL_ADC_PollForConversion()` waits up to **10 ms** for it to complete, and `HAL_ADC_GetValue()` returns the **12-bit** result (range `0–4095`).
- `Custom_STM_App_Update_Char(CUSTOM_STM_ADC_VALUE_CHARACTERISTIC, (uint8_t *)&adcValue)` — the CubeMX-generated helper that updates the characteristic value and triggers a BLE notify to any subscribed peer. Since `adcValue` is a `uint16_t` (2 bytes), the characteristic's `Value Length = 2` matches.
- `printf("%u\r\n", adcValue)` echoes the value to the **USART1** (115200 baud) so you can watch it on a serial terminal in parallel.
- `HAL_ADC_Stop()` releases the ADC until the next 1-second tick.

> **Note:** `MX_APPE_Process()` (inside the `USER CODE END WHILE` block) must keep running each loop iteration — it services the BLE stack. The 1-second gated block sits **after** it, inside `USER CODE BEGIN 3`.

Rebuild and reflash.

## Step 5 — Test with ST BLE Toolbox

1. Install **ST BLE Toolbox** on Android or iOS.
2. Scan — your device `ADC-STM32` should appear.
3. Connect to the device.
4. Open the **ADC_Service** service.
5. You will see **ADC_Value_Characteristic** — its value should update automatically every ~100ms with the current ADC reading.

## Troubleshooting

### Device not appearing in scan
- Confirm the FUS and BLE stack are flashed and **Start Wireless Stack** succeeded.
- Confirm `AD_TYPE_COMPLETE_LOCAL_NAME` is set to `ADC-STM32`.
- Confirm `RF1` is activated in the RF configuration.

### Notify not updating
- Verify the GATT characteristic is configured with **NOTIFY** property (not WRITE).
- Check that the characteristic handle used in the notify call matches `ADC_Value_Characteristic`.
- Ensure `notify_buf` is populated with the correct ADC value before calling the notify API.

### ADC reading seems wrong
- Check that the correct pin/channel is configured in CubeMX.
- Verify ADC clock prescaler and resolution settings.
- Try printing the raw ADC value to a UART or LED to confirm hardware is reading correctly.
